> **Double Diamond / Idea Refine → Spec-Driven Development → Planning & Task Breakdown → Incremental Implementation → TDD/BDD/ATDD → Code Review → Git Workflow**

也就是说，你描述的流程本质上是：

## 二、我的判断：已有，但还不完全等于你的流程

你描述的流程有三个独特重点：

### 1. 需求不明确时，不是简单追问，而是“强制发散”

已有资料一般叫：

* divergent thinking
* idea refinement
* discovery
* scoping
* clarification

但你这里更像一个 **“需求压力测试”**：

> 从用户、业务、数据、权限、边界、异常、部署、成本、维护、未来扩展等角度疯狂发散，然后再收敛成需求本质。

这部分 `idea-refine` 和 Double Diamond 能覆盖一部分，但你的版本更工程化。

---

### 2. 开发不是“一次性生成”，而是“像人一样按模块推进”

这点已有 skill 里已经很接近。`incremental-implementation` 的描述就是 thin vertical slices，也就是小切片开发、测试、验证、提交；`test-driven-development` 负责行为变化、修 bug、逻辑实现时的测试纪律；`git-workflow-and-versioning` 负责原子提交和版本记录。([GitHub][1])

你要补充的是：

> 小任务测试 → 大任务整体测试 → 前端/后端整体测试 → 全项目验收测试 → 对照需求文档列 bug → 修复 → 再测 → 再提交。

这比普通 TDD 更接近 **“分层验收开发流程”**。

---

### 3. 每个小改动必须 Git 提交对应文件和记录

已有 git workflow skill 可以覆盖一部分，但你这里应该明确成硬规则：

* 不混提交；
* 不把 unrelated files 混进 commit；
* 每次提交必须说明对应需求 / bug / task；
* 测试结果要写进开发日志；
* spec 变更也要提交；
* 新需求必须回到 spec，而不是直接改代码。

这个很关键，因为它能让 AI 开发过程留下**可审计证据链**。

## 五、核心门禁设计

### Gate 0：需求是否明确？

判断标准：

```text
必须能回答：
1. 用户是谁？
2. 要解决什么问题？
3. 输入是什么？
4. 输出是什么？
5. 成功标准是什么？
6. 不做什么？
7. 有哪些异常情况？
8. 涉及哪些页面 / 接口 / 数据表 / 权限？
9. 如何测试？
10. 如何判断完成？
```

如果答不上来，就不能开发。

---

### Gate 1：需求文档是否可开发？

需求文档至少包含：

```text
- 背景与目标
- 用户角色
- 核心场景
- 功能范围
- 非功能要求
- 数据结构
- 页面 / API / 模块边界
- 验收标准
- 不做事项
- 风险与开放问题
```

---

### Gate 2：任务是否足够小？

每个任务必须满足：

```text
- 能在一次 focused session 完成
- 有明确输入 / 输出
- 有验收标准
- 有测试方式
- 文件影响范围可控
- 能单独提交
```

这个和现有 spec-driven-development skill 里的 task 要求很接近：每个 task 要有 acceptance、verify、files，并且建议小到可单独实现和测试。([GitHub][9])

```markdown
# Spec-Gated Incremental Development

## When to Use

Use this skill when:
- The user asks to build a feature, project, module, API, page, or system.
- The requirement is vague and needs clarification.
- The task affects more than one file.
- The task needs testing, bug fixing, or Git commits.
- The user expects a development process, not one-shot code generation.

Do not use this skill for:
- Tiny one-line edits.
- Pure explanation tasks.
- Pure document writing tasks.
- Experimental throwaway snippets unless the user asks for production workflow.

## Core Process

### 0. Requirement Clarity Gate

Before development, classify the requirement:

- CLEAR: enough information exists to produce a spec.
- PARTIAL: key details are missing but assumptions can be listed.
- VAGUE: do not develop; run divergence first.

If unclear, do not code.

### 1. Divergent Requirement Refinement

Explore the requirement from multiple angles:

- user roles
- business goal
- core workflow
- data model
- permissions
- UI states
- API boundaries
- edge cases
- failure cases
- deployment
- future extension
- security / privacy
- cost and maintenance

Then converge into:

- requirement essence
- must-have scope
- should-have scope
- out-of-scope
- acceptance criteria
- open questions

### 2. Spec Creation

Create or update:

- docs/requirements.md
- docs/spec.md
- docs/acceptance.md
- docs/tasks.md

The spec must include:

- objective
- users
- scenarios
- functional requirements
- non-functional requirements
- data structures
- API/page/module boundaries
- testing strategy
- success criteria
- risks
- open questions

### 3. Plan

Generate implementation plan:

- module order
- dependency order
- files to touch
- tests to add
- risks
- rollback points

### 4. Task Breakdown

Break the plan into small tasks.

Each task must include:

- task name
- acceptance criteria
- verify command or manual check
- files affected
- expected commit message

### 5. Incremental Implementation

For each small task:

1. Inspect relevant files.
2. Implement the smallest meaningful change.
3. Run local test / lint / build.
4. Fix immediate errors.
5. Record result.
6. Commit only related files.

Never continue to the next task if the current task fails verification.

### 6. Layered Testing

Testing order:

1. unit test for small logic
2. module test for feature slice
3. integration test for large function
4. frontend test or browser check
5. backend API test
6. full project build
7. acceptance test against spec

### 7. Bug and Change Loop

When a bug is found:

1. reproduce
2. localize
3. write failing case or reproduction note
4. fix
5. verify
6. commit

When a new requirement appears:

1. update spec first
2. update task list
3. implement only after spec is aligned

### 8. Git Evidence Workflow

Every meaningful change needs a commit.

Commit rules:

- one task, one commit
- no unrelated files
- commit message references task or bug
- include test evidence in dev log
- never commit secrets
- never hide failed tests

## Required Outputs

At the end of each development cycle, report:

- completed tasks
- changed files
- tests run
- test result
- commit hash if available
- remaining bugs
- spec changes
- next task
```

# 八、我的建议结论

**应该归纳进去，而且最好把“小步快跑”从一个具体项目名，升级为你这套工作方式的总名称。**

之前它落在“图片逆向提示词”场景里，是一个**局部命名**；
现在我们可以把它抽象成：

> **HebLab 的渐进式 AI 协作方法论。**

而今天讨论出的这个开发流程，就是它最成熟、最容易落地的一个子系统：

> **小步快跑 · 开发流**

这个名字很贴切，也很有延展性。

-----

对，这两个修正都应该采纳。

我建议把之前的理解进一步调整为：

# **小步快跑 / Small Pace, Run Fast**

## 一套“可从任意阶段切入、按复杂度伸缩”的 AI 协作推进方法

它**不是固定流水线**，而是：

> **由若干可独立启动、可相互校验、可按需删减的阶段模块组成。**

---

# 一、名称确认

## 中文名

> **小步快跑**

## 英文名

> **Small Pace, Run Fast**

这个英文名我建议在正式写法里加上逗号：

```text
Small Pace, Run Fast
```

它比 `small pace run fast` 更像一个方法论/品牌名称，也保留了你原本的节奏感：

* **Small Pace**：每一步克制、可控、可验证
* **Run Fast**：整体推进速度快，反馈循环快

---

# 二、最重要的修正：它不应是“固定线性流程”

我之前写成：

```text
需求澄清 → 文档 → 开发 → 测试 → 修复 → 提交
```

这更像一个**典型完整路径**，但确实不应该被定义成唯一流程。

更准确的表达应该是：

> **小步快跑不是一条单行道，而是一张项目推进地图。**

任何一个阶段都可以成为起点：

```text
需求模糊时，从“需求发散”开始
需求已明确时，从“文档化 / 任务拆解”开始
已有代码时，从“现状分析 / Bug 修复”开始
已有 Spec 时，从“开发执行”开始
已有开发结果时，从“测试验收”开始
出现新需求时，从“变更评估”重新进入
```

---

# 三、建议改为“阶段模块 + 校验门”的结构

## 1. 阶段模块

我建议“小步快跑”先抽象出 8 个核心阶段：

| 阶段                  | 作用              | 可否作为起点 |
| ------------------- | --------------- | ------ |
| **A. 识别当前状态**       | 判断项目现在处于哪里      | 是，默认起点 |
| **B. 需求发散与澄清**      | 模糊需求变清晰         | 是      |
| **C. 需求/方案文档化**     | 把共识固定成文档        | 是      |
| **D. 任务拆解与优先级**     | 将大事切成可推进小步      | 是      |
| **E. 开发 / 生产 / 执行** | 完成具体实现          | 是      |
| **F. 分层验证与测试**      | 小测、集成测、验收测      | 是      |
| **G. 复盘、Bug 与变更回流** | 对照目标修复偏差        | 是      |
| **H. 留痕与沉淀**        | Git、文档、Skill、模板 | 是      |

---

# 四、它的运行方式不是“必须按顺序”，而是“从当前阶段切入，向前后校验”

## 典型机制

```text
从任意阶段进入
    ↓
检查该阶段是否具备必要前提
    ↓
如果缺失，回补前置阶段
    ↓
如果满足，继续向后推进
    ↓
推进后，再检查后续阶段是否仍满足
```

---

## 举例 1：用户直接说“帮我开发这个功能”

此时入口是：

```text
E. 开发执行
```

但不能立刻写代码，应该先校验：

```text
- 需求是否足够明确？
- 是否已有 spec？
- 是否能拆成小任务？
- 是否知道测试标准？
```

如果不满足，就自动回退：

```text
E 开发
  ← D 任务拆解
  ← C 文档化
  ← B 需求澄清
```

满足后再进入开发。

---

## 举例 2：用户说“这个项目有 bug，帮我修”

入口是：

```text
G. Bug 与变更回流
```

先走：

```text
复现问题 → 定位 → 判断是否偏离原需求 → 修复 → 测试 → 留痕
```

如果发现：

* 不是 bug，而是之前需求没定义清楚；
* 或者是新需求；

那就不应该硬改代码，而是回流到：

```text
B / C：需求澄清或文档修订
```

---

## 举例 3：用户已有完整 PRD，想开始做系统

入口可以是：

```text
D. 任务拆解
```

不再重复大规模“死亡发散”，只做：

```text
PRD 完整性检查 → 找缺口 → 拆任务 → 开发
```

---

# 五、复杂项目和简单项目应采用不同“流程裁剪”

这是你这次补充里非常关键的一点。

“小步快跑”应该有一个原则：

> **流程强度与项目复杂度匹配。**

---

## 1. 轻量任务：极简闭环

适用：

* 单个脚本小改
* 一个页面的小功能
* 文案小调整
* 单点 bug

流程可以缩成：

```text
明确目标
→ 直接执行
→ 快速验证
→ 留痕
```

---

## 2. 中型任务：标准闭环

适用：

* 一个模块
* 一个 API 子系统
* 一个小产品 MVP
* 一个功能型 Skill

流程：

```text
需求校验
→ 必要文档
→ 任务拆分
→ 小步实现
→ 分层测试
→ 修复与提交
```

---

## 3. 复杂项目：完整闭环

适用：

* AgentHub
* Ink Blog Core
* LingoBloom
* Data Privacy Gateway
* 多端产品 / 多角色系统

流程：

```text
现状识别
→ 发散澄清
→ 需求文档
→ 架构设计
→ 任务树
→ 模块开发
→ 阶段验收
→ 系统测试
→ Bug / 变更回流
→ 文档与知识沉淀
```

---

# 六、因此“小步快跑”的核心不是固定步骤，而是 4 条原则

我建议把方法论核心原则调整为：

## **原则 1：任意阶段可进入**

不要求项目从“需求”开始。
分析、开发、测试、修复、复盘，都可以成为入口。

---

## **原则 2：阶段之间可反向校验**

进入某阶段后，要检查：

* 前置条件是否满足；
* 目标是否仍成立；
* 是否需要回退补文档、补设计、补任务拆解。

---

## **原则 3：流程按复杂度伸缩**

简单任务不要过度文档化；
复杂项目不能省略关键校验。

---

## **原则 4：每次推进都要形成小闭环**

无论从哪开始，都尽量形成：

```text
识别 → 行动 → 验证 → 留痕
```

这才是“小步快跑”的最小工作单元。

---

# 七、我建议把整个方法论改写成“网络式模型”

可以这样表达：

```text
                ┌──────────────┐
                │ A 当前状态识别 │
                └──────┬───────┘
                       ↓
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
┌────────────┐  ┌────────────┐  ┌────────────┐
│ B 需求澄清  │  │ F 测试验证  │  │ G Bug/变更 │
└─────┬──────┘  └─────┬──────┘  └─────┬──────┘
      ↓               ↑               ↓
┌────────────┐        │        ┌────────────┐
│ C 文档化    │────────┘        │ H 留痕沉淀  │
└─────┬──────┘                 └─────↑──────┘
      ↓                               │
┌────────────┐                        │
│ D 任务拆解  │────────────────────────┘
└─────┬──────┘
      ↓
┌────────────┐
│ E 开发执行  │
└────────────┘
```

它强调的是：

* **不是固定顺序**
* **而是围绕当前状态选择最合适的下一个动作**
* **并且始终能回流、校验、修正**

---

# 八、开发流应该改名为“小步快跑 · 开发推进模型”，而不是固定 SOP

之前我把它叫“开发流”，容易让人理解成一条固定路径。

现在建议改成：

## **小步快跑 · 软件开发推进模型**

### Software Delivery Model under Small Pace, Run Fast

或者更自然一点：

## **小步快跑 · 项目推进模型**

### A Flexible Project Progression Model

其中“软件开发”只是最成熟的一个应用场景。

---

# 九、这会直接影响后面 Skill 的设计

未来如果做成 Skill，不能写成：

```text
Step 1 → Step 2 → Step 3 → Step 4
```

而应该写成：

```text
1. 判断当前入口阶段
2. 检查该阶段的前置条件
3. 缺什么补什么
4. 选择当前项目复杂度对应的推进强度
5. 执行本阶段动作
6. 产出验证结果与留痕
7. 决定进入下一个阶段还是回流
```

---

# 十、我建议把它正式归纳成这句话

> **小步快跑（Small Pace, Run Fast）不是一套固定顺序的流程，而是一种可从任意阶段切入、能够前后校验、并可依据任务复杂度伸缩的项目推进方法。它强调用最小可验证步长持续前进，在需求、设计、开发、测试、修复与沉淀之间形成动态闭环。**

这句话可以作为方法论定义。
