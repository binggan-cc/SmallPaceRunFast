# 小步快跑 Small Pace, Run Fast 方法论 v1.0

## 0. 文档定位

**中文名称：** 小步快跑

**英文名称：** Small Pace, Run Fast

**文档版本：** v1.0

**文档性质：** 方法论定义文档 / 项目推进框架 / AI 协作工作流基础原则

---

## 1. 方法论定义

### 1.1 一句话定义

> **小步快跑（Small Pace, Run Fast）是一种可从任意阶段切入、能够前后校验、并可依据任务复杂度伸缩的项目推进方法。它强调用最小可验证步长持续前进，在需求、设计、开发、测试、修复与沉淀之间形成动态闭环。**

### 1.2 更完整的解释

“小步快跑”并不等同于单纯追求速度，也不是把一个项目机械拆成固定流水线。它的核心并不是“赶紧做完”，而是：

1. **在不确定中先建立足够清晰的判断；**
2. **将复杂问题拆成可验证、可推进的小步；**
3. **每推进一步，都形成反馈、验证与留痕；**
4. **当发现前提不成立、目标偏移或新信息出现时，可以及时回流到更合适的阶段；**
5. **根据任务复杂程度选择适当流程强度，既避免轻任务过度流程化，也避免复杂项目无控制地裸奔。**

因此，小步快跑是一套面向真实工作场景的**动态推进模型**，而不是线性的固定 SOP。

## 2. 方法论的产生背景

### 2.1 从具体项目经验中抽象而来

“小步快跑”最早可以用于描述视觉提示词、图片逆向提示词等项目中的迭代方式：

* 先观察已有结果；
* 再拆解其中的主体、构图、风格、材质、光线与控制项；
* 生成初版提示词；
* 出图验证；
* 对比偏差并修订；
* 逐步沉淀出更稳定的提示词结构与可复用模板。

这种工作方式不是一次成型，而是通过**小步生成、小步对照、小步修正**不断逼近目标。

### 2.2 推广到软件开发与复杂项目推进

在后续的软件开发、Agent 系统设计、产品方案设计、文档编制与能力沉淀过程中，同样形成了一种稳定偏好：

* 需求不清时，先发散，不急于开发；
* 一旦本质明确，就要及时收敛并形成文档；
* 开发不以“一次性大块完成”为目标，而是按模块、按任务推进；
* 每个小任务完成后及时验证；
* 一个功能阶段完成后进行更大范围的集成测试；
* 最终对照需求文档检查偏差；
* Bug 修复、新需求调整与文档更新构成回流；
* 每一次有效推进都需要形成版本记录、开发记录或方法沉淀。

从这些经验可以抽象出“小步快跑”的通用方法论。

## 3. 核心思想

### 3.1 不是盲目快，而是“可控地快”

“小步快跑”的“快”，不是跳过思考、跳过验证、跳过文档，而是：

* **减少错误方向上的大投入；**
* **减少一次性重做的代价；**
* **缩短反馈周期；**
* **用更短的验证回路形成更高的整体推进速度。**

它追求的是**系统层面的推进效率**，不是单点动作的表面速度。

### 3.2 不是流程崇拜，而是状态驱动

“小步快跑”并不要求所有项目都从“需求澄清”开始，也不要求所有任务都必须经历完全相同的过程。

它主张：

> **先判断当前所处状态，再选择最合适的下一步。**

例如：

* 新项目但需求模糊：从需求发散开始；
* PRD 已完整：从任务拆解开始；
* 已有代码但问题频出：从测试与 Bug 回流开始；
* 有成熟功能但缺少归档：从留痕与沉淀开始；
* 设计方案已定：直接进入小步实现，但要先检查前置条件是否满足。

### 3.3 每一步都应是“最小可验证推进”

“小步”不是把任务拆碎到没有意义，而是将任务切成一个个：

* 有明确目标；
* 有可见产出；
* 有验证方式；
* 有失败回退空间；
* 有记录价值；
* 能推动整体向前。

这样的单元，才构成“小步快跑”的基本推进粒度。

## 4. 四项核心原则

### 原则一：任意阶段可进入

“小步快跑”不是固定从 1 到 N 的流程，而是一个可从不同节点切入的阶段网络。

常见入口包括：

* 需求澄清；
* 方案设计；
* 任务拆解；
* 开发执行；
* 测试验收；
* Bug 修复；
* 新需求调整；
* 复盘沉淀。

**要求：**

进入某个阶段后，不是机械向后走，而是先检查该阶段是否具备必要前提；若不具备，应补足缺失环节。

---

### 原则二：阶段之间可前后校验

每个阶段既要服务于后续推进，也要反向校验前序判断是否可靠。

例如：

* 开发阶段发现需求边界不清，应回流到需求或文档阶段；
* 测试阶段发现设计遗漏，应回流到方案设计；
* Bug 修复阶段发现其实是新需求，应进入变更评估，而不是直接硬改；
* 复盘阶段发现之前没有形成可维护文档，应补足文档沉淀。

**要求：**

每个阶段都必须具备“继续推进”与“回流修正”两种能力。

---

### 原则三：流程强度按复杂度伸缩

“小步快跑”拒绝两种极端：

1. **简单问题过度流程化；**
2. **复杂问题过度随意化。**

因此，需要根据任务复杂度选择不同强度的推进模式。

#### 轻量任务

适用于：

* 单点修改；
* 文案微调；
* 小脚本补丁；
* 简单 Bug；
* 单一页面小改动。

建议流程：

> 明确目标 → 快速执行 → 立即验证 → 简要留痕

#### 中型任务

适用于：

* 单个功能模块；
* 小型系统子模块；
* 一组页面与接口联动；
* 一个可独立交付的 Skill；
* 一条较完整的业务流。

建议流程：

> 需求校验 → 必要文档 → 任务拆分 → 小步实施 → 阶段测试 → 修复与提交

#### 复杂项目

适用于：

* 多角色、多模块系统；
* 长周期产品研发；
* 需要架构设计与阶段验收的工程；
* 涉及权限、安全、数据、部署的系统；
* 多 Agent 协作项目。

建议流程：

> 现状识别 → 发散澄清 → 需求文档 → 架构设计 → 任务树 → 模块开发 → 分层测试 → 系统验收 → Bug/变更回流 → 沉淀归档

---

### 原则四：每次推进都要形成小闭环

无论从哪个阶段开始，无论任务规模大小，都应尽量形成如下最小闭环：

> **识别 → 行动 → 验证 → 留痕**

#### 识别

* 当前问题是什么；
* 当前处在哪个阶段；
* 现有信息是否足够；
* 风险在哪里。

#### 行动

* 做一个足够小但有意义的推进动作；
* 不追求一次性完成全部；
* 优先完成最能降低不确定性的部分。

#### 验证

* 结果是否符合预期；
* 是否满足阶段目标；
* 是否暴露新的前提问题；
* 是否需要回流。

#### 留痕

* 记录决策；
* 更新文档；
* 保留测试结果；
* Git 提交；
* 归档经验；
* 沉淀为 Skill / 模板 / 工作流。

---

## 5. 阶段网络模型

“小步快跑”推荐采用“阶段模块 + 校验门”的组织方式。

### 5.1 八个核心阶段

| 编号 | 阶段           | 作用                 | 可否作为起点 |
| -- | ------------ | ------------------ | ------ |
| A  | 当前状态识别       | 判断项目所处位置、目标与不确定性   | 是      |
| B  | 需求发散与澄清      | 将模糊诉求展开、分析、归纳      | 是      |
| C  | 需求 / 方案文档化   | 将共识固化为可讨论、可执行文档    | 是      |
| D  | 任务拆解与优先级     | 将大目标拆成可执行小步        | 是      |
| E  | 开发 / 生产 / 执行 | 完成具体实现动作           | 是      |
| F  | 分层验证与测试      | 小测、集成测、整体验收        | 是      |
| G  | Bug、偏差与变更回流  | 处理不符合、遗漏和新需求       | 是      |
| H  | 留痕与沉淀        | Git、文档、知识、模板、Skill | 是      |

---

## 6. 任意阶段切入机制

### 6.1 通用规则

当一个任务从任意阶段进入时，遵循以下判断：

1. **识别当前入口阶段；**
2. **检查该阶段需要的前置条件是否满足；**
3. **若不满足，回补必要阶段；**
4. **若满足，执行当前阶段；**
5. **执行后检查是否进入下一阶段，或需要回流；**
6. **完成本轮闭环并留痕。**

---

### 6.2 典型切入场景

#### 场景一：从模糊需求切入

用户说：

> 我想做一个 AI 内容工具，但还没完全想清楚。

起点：

> B. 需求发散与澄清

推进方式：

1. 拆解目标用户、使用场景、输入输出、价值主张；
2. 发散不同形态、不同复杂度、不同商业化方式；
3. 收敛到最值得做的核心方向；
4. 形成需求文档；
5. 进入任务拆解或方案设计。

---

#### 场景二：从已有 PRD 切入

用户说：

> 需求文档已经定了，帮我开始拆开发任务。

起点：

> D. 任务拆解与优先级

推进方式：

1. 快速检查 PRD 是否存在关键缺口；
2. 如缺口不影响推进，可标注假设后继续；
3. 拆成阶段任务、小任务、验收标准；
4. 进入执行阶段。

---

#### 场景三：从开发执行切入

用户说：

> 按这个设计直接开始写代码。

起点：

> E. 开发 / 执行

但要先校验：

* 需求是否明确；
* 方案是否完整；
* 是否知道要先做哪一块；
* 是否明确测试方式。

若不满足，应回到 C 或 D；若满足，则直接按小步推进。

---

#### 场景四：从 Bug 修复切入

用户说：

> 这个功能报错了，帮我修。

起点：

> G. Bug、偏差与变更回流

推进方式：

1. 复现问题；
2. 确认问题类型：Bug / 需求遗漏 / 新需求；
3. 若是 Bug，直接进入修复；
4. 若是需求遗漏，回流到文档阶段；
5. 若是新需求，进入变更评估；
6. 修复后测试并留痕。

---

#### 场景五：从测试验收切入

用户说：

> 帮我对照需求检查这个项目是否完成。

起点：

> F. 分层验证与测试

推进方式：

1. 对照需求文档建立验收清单；
2. 逐项验证；
3. 输出完成项、缺失项、Bug、风险项；
4. 缺失或偏差进入 G 阶段；
5. 完成后更新 H 阶段沉淀。

## 7. 阶段详解

### A. 当前状态识别

#### 目标

判断项目当前所处阶段，以及最适合从哪里切入。

#### 主要问题

* 现在是想法、文档、代码、测试还是复盘阶段？
* 当前最不确定的是什么？
* 当前最阻碍推进的是什么？
* 现在做什么动作，最能降低整体不确定性？

#### 典型输出

* 当前阶段判断；
* 建议入口；
* 缺失信息清单；
* 推进强度建议：轻量 / 中型 / 复杂。

---

### B. 需求发散与澄清

#### 目标

当需求不明确时，通过多角度发散，帮助识别真正问题，并收敛出可执行需求。

#### 可发散的维度

* 用户是谁；
* 核心痛点是什么；
* 解决的是效率问题、体验问题还是能力问题；
* 现有替代方案是什么；
* 为什么现在值得做；
* 输入是什么；
* 输出是什么；
* 是否存在角色权限差异；
* 异常情况有哪些；
* 边界在哪里；
* 成功标准是什么；
* 后续扩展方向是什么；
* 是否涉及数据安全、成本、部署、维护。

#### 收敛产出

* 需求本质；
* 核心目标；
* Must Have / Should Have / Later；
* 非目标；
* 开放问题；
* 初步验收标准。

---

### C. 需求 / 方案文档化

#### 目标

将需求、方案和关键约束固化为后续可依赖的共识载体。

#### 文档化内容可以根据复杂度裁剪

##### 轻量任务

* 目标说明；
* 变更点；
* 简单验收条件。

##### 中型任务

* 背景；
* 目标；
* 功能范围；
* 页面 / 接口 / 数据点；
* 验收标准；
* 风险。

##### 复杂项目

* PRD；
* 架构设计；
* 模块说明；
* 数据模型；
* API 契约；
* 页面流；
* 权限与异常；
* 测试策略；
* 里程碑；
* 版本计划。

---

### D. 任务拆解与优先级

#### 目标

将可执行方向拆分成可推进的小任务。

#### 一个合格的小任务应具备

* 明确目标；
* 清晰边界；
* 对应模块；
* 验收标准；
* 测试方式；
* 相关文件或影响范围；
* 可单独交付；
* 可单独记录或提交。

#### 常见拆分方式

* 按用户流程拆；
* 按系统模块拆；
* 按数据流拆；
* 按前后端拆；
* 按风险优先级拆；
* 按 MVP 与增强项拆。

---

### E. 开发 / 生产 / 执行

#### 目标

以小步推进的方式完成实际产出。

#### 软件开发场景中的执行原则

1. 一次只解决一个明确问题；
2. 每次修改尽量保持影响面可控；
3. 先做最小闭环；
4. 不在一个任务中混入多种不相关变更；
5. 发现前提不成立时及时停下回流，而不是继续堆代码；
6. 复杂问题优先做验证性实现，再做完整实现。

#### 视觉提示词场景中的执行原则

1. 一次只调整一个或少数几个明确变量；
2. 明确本轮是调构图、风格、主体、光影还是细节；
3. 输出后立即与目标比对；
4. 避免在没有对比结论的情况下连续盲改。

---

### F. 分层验证与测试

#### 目标

通过由小到大的验证体系，确认当前推进是真正有效的。

#### 软件开发中的测试层次

1. 单点验证：一个函数、一个组件、一个接口；
2. 小模块测试：若干逻辑构成的独立功能；
3. 大功能集成测试：完整业务链路；
4. 前端整体测试；
5. 后端整体测试；
6. 全项目级验收测试；
7. 对照需求文档检查缺项、错项、偏项。

#### 视觉提示词中的验证层次

1. 是否保留目标主体；
2. 是否匹配构图；
3. 是否匹配风格；
4. 是否匹配材质与氛围；
5. 是否稳定复现；
6. 是否可模板化。

---

### G. Bug、偏差与变更回流

#### 目标

将错误、遗漏和变化转化为可管理的推进对象。

#### 需区分三类问题

| 类型   | 含义          | 处理方式             |
| ---- | ----------- | ---------------- |
| Bug  | 结果不符合既定需求   | 修复并测试            |
| 需求遗漏 | 文档没有覆盖但本应覆盖 | 回流补文档，再修正        |
| 新需求  | 新增目标或新范围    | 做变更评估，决定是否纳入当前版本 |

#### 标准动作

* 描述问题；
* 尝试复现；
* 确认归类；
* 修复或回流；
* 验证结果；
* 记录变化。

---

### H. 留痕与沉淀

#### 目标

让每一次推进都变成可回看、可复用、可继承的资产。

#### 沉淀对象

* Git 提交；
* 变更记录；
* 测试记录；
* 决策记录；
* 文档更新；
* Prompt 模板；
* 复盘总结；
* Skill；
* 工作流；
* 规范。

#### 软件开发中的 Git 留痕建议

* 一个小任务尽量对应一个提交；
* 不混入无关文件；
* 提交说明应能看出对应需求、Bug 或改动目标；
* 文档变化与代码变化应保持一致；
* 不应只留最终结果，而要保留关键过程证据。

---

## 8. 小步快跑的最小工作单元

无论任务大小，都可以压缩成如下基本单元：

```text
识别当前问题
→ 选择一个最小推进动作
→ 执行该动作
→ 验证结果
→ 判断继续、回流或结束
→ 留痕
```

这就是“小步快跑”的最小动作循环。

## 9. 在软件开发场景中的落地

### 9.1 典型完整流程示例

```text
需求模糊
→ 发散分析
→ 收敛成需求文档
→ 方案设计
→ 任务拆解
→ 小任务开发
→ 小任务测试
→ 小任务 Git 提交
→ 多任务组成大功能
→ 大功能集成测试
→ 前后端整体测试
→ 项目级验收
→ Bug / 新需求回流
→ 修复与再验证
→ 文档与经验沉淀
```

### 9.2 但这不是唯一固定路径

同一个方法论也支持：

* 从现有代码分析切入；
* 从测试验收切入；
* 从 Bug 修复切入；
* 从文档复核切入；
* 从版本迭代切入。

### 9.3 软件开发中的七个建议门禁

1. 是否知道当前任务真正目标；
2. 是否知道当前任务边界；
3. 是否知道怎么判断完成；
4. 是否知道本轮最小可推进动作；
5. 是否能在完成后立即验证；
6. 是否需要更新文档；
7. 是否需要形成提交与记录。

---

## 10. 在视觉提示词与图片逆向中的落地

### 10.1 典型完整流程示例

```text
观察目标图像
→ 拆解主体、构图、风格、材质、光影、镜头
→ 生成逆向描述
→ 形成初版提示词
→ 出图验证
→ 对比差异
→ 局部修订
→ 再次生成
→ 沉淀稳定模板
```

### 10.2 小步快跑在视觉工作中的意义

它避免两种常见问题：

1. 一上来就写超长提示词，但不知道真正控制了什么；
2. 连续盲改，没有结构化反馈，最终无法复现与沉淀。

### 10.3 推荐迭代维度

* 主体准确性；
* 版式与构图；
* 透视与镜头；
* 风格语义；
* 材质与光影；
* 色彩系统；
* 细节密度；
* 文本排除项；
* 可复用模板结构。

---

## 11. 在内容生产场景中的落地

虽然“小步快跑”最初更容易在开发和视觉生成中体现，但它同样适用于内容生产。

### 11.1 内容生产示例

```text
选题模糊
→ 发散主题角度
→ 确定核心立场
→ 搭建结构
→ 先写最关键段落
→ 局部审读
→ 扩写全文
→ 统一风格
→ 校验论证与例子
→ 形成配图建议、标题、摘要与结尾引导
→ 沉淀为可复用写作模板
```

### 11.2 内容工作中的小步原则

* 先明确“文章到底要说什么”；
* 再决定结构，不要直接铺字数；
* 先写最关键段落，验证文章成立；
* 每次修改优先解决一个主要问题；
* 完成后形成模板、风格规则或可复用经验。

---

## 12. 在 Agent 与复杂工作流中的落地

“小步快跑”也可作为 AI Agent 工作流设计原则：

* 先判断任务状态；
* 再选择当前最值得执行的环节；
* 不让 Agent 无边界地“一把梭”；
* 每一步都有局部产出与局部验证；
* 复杂任务可动态回流；
* 最后形成可追踪日志和资产沉淀。

### 12.1 Agent 工作流中的典型结构

```text
理解任务
→ 判断任务清晰度
→ 选择阶段入口
→ 调用合适技能
→ 产生阶段性产物
→ 做阶段校验
→ 进入下一阶段或回流
→ 最终交付与沉淀
```

## 13. 小步快跑与常见方法的关系

“小步快跑”与一些成熟方法存在相通之处，但并不完全等同。

| 方法                      | 相通点        | 差异                    |
| ----------------------- | ---------- | --------------------- |
| 敏捷开发                    | 迭代、反馈、持续交付 | 小步快跑更强调任意阶段切入与前后回流    |
| 精益创业                    | 小投入验证假设    | 小步快跑不只用于商业验证，也用于工程与内容 |
| 双钻模型                    | 发散与收敛      | 小步快跑覆盖需求之后的执行、验证、沉淀   |
| TDD / BDD               | 测试驱动与验收    | 小步快跑测试更广义，不局限编码       |
| Spec-Driven Development | 文档先行与规格对齐  | 小步快跑允许从任意阶段进入，并按复杂度裁剪 |

## 14. 方法论适用边界

### 14.1 适用场景

* 复杂项目设计；
* 软件开发；
* Agent 工作流；
* AI 产品设计；
* 文档 / PRD / 架构方案；
* 视觉提示词生成与逆向；
* 长周期内容生产；
* 需要高频校验与过程沉淀的任务。

### 14.2 不适合过度使用的场景

* 纯粹一次性的简单闲聊；
* 不需要校验的小问题；
* 完全探索性、没有目标的自由创作；
* 价值极低、无需形成任何沉淀的微小动作。

## 15. 推荐表达口径

### 15.1 方法论简介版

> **小步快跑（Small Pace, Run Fast）是一套面向复杂任务与 AI 协作场景的动态推进方法。它不要求项目沿固定流程前进，而是允许从需求、设计、执行、测试或复盘中的任意阶段切入，并通过前后校验与复杂度裁剪形成最合适的推进路径。其核心是用最小可验证步长持续前进，在每一次行动后及时验证、回流和留痕，从而实现真正可控的快速推进。**

### 15.2 更短的品牌化表达

> **小步，不是慢；快跑，不是乱。**
> **先找到当前最值得推进的一步，再把每一步都走成闭环。**

### 15.3 适合放在项目首页的一句话

> **Small Pace, Run Fast：从任意阶段切入，用最小可验证步长推进复杂项目。**

## 16. 后续可继续扩展的方向

后续可以继续把“小步快跑”发展为：

1. **方法论白皮书**：对外阐述完整理念；
2. **项目推进 Skill**：用于 AI 协作过程控制；
3. **开发流 Skill**：针对软件开发做更细执行规范；
4. **视觉提示词 Skill**：针对图片逆向、视觉 Prompt OS 落地；
5. **内容生产 Skill**：针对公众号文章、知识视觉、研究型内容；
6. **Agent 编排框架**：使 Agent 能基于阶段判断自动推进任务；
7. **流程图与操作手册**：方便团队或未来系统落地。

## 17. v1.0 总结

“小步快跑”不是一句效率口号，而是一种对复杂任务更稳健的推进哲学：

* 不迷信一次性想清楚一切；
* 也不赞成在混乱中仓促行动；
* 它承认现实工作往往从半成品、半信息和不完整状态出发；
* 因而更重视当前状态判断、局部推进、阶段验证与回流修正；
* 最终用一个个可验证的小闭环，推动复杂目标持续向前。

> **小步，是为了每一步都站得住；快跑，是因为每一步都不白走。**
