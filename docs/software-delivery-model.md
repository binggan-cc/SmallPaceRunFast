# 小步快跑 · 软件开发推进模型 v1.0

## 0. 文档定位

**中文名称：** 小步快跑 · 软件开发推进模型

**英文名称：** Software Delivery Model under Small Pace, Run Fast

**文档版本：** v1.0

**文档性质：** 软件开发流程文档 / AI 协作开发规范 / Skill 设计基础

**上位方法论：** 小步快跑 Small Pace, Run Fast

---

## 1. 一句话定义

> **小步快跑 · 软件开发推进模型，是一套从任意阶段切入、以需求明确性为门禁、以小任务实现和分层验证为推进方式，并通过文档、测试和 Git 形成证据链的 AI 协作开发方法。**

它不是固定线性 SOP，而是“小步快跑”方法论在软件开发场景中的落地模型。

它的最小动作循环是：

```text
识别当前状态
→ 选择最小可验证动作
→ 执行
→ 验证
→ 判断继续、回流或结束
→ 留痕
```

---

## 2. 模型要解决的问题

AI 协作开发中常见的问题不是“写代码太慢”，而是：

* 需求还不清楚就开始实现；
* 一次性生成大块代码，难以验证、定位和回滚；
* 任务拆得太粗，导致每一步都不可独立验收；
* 小任务完成后没有及时测试，问题堆到集成阶段才暴露；
* Bug、需求遗漏和新需求混在一起处理；
* 代码、文档、测试结果和 Git 提交之间缺少对应关系；
* 最终交付只看到结果，看不到判断过程和证据链。

因此，该模型强调：

```text
需求过门禁
→ 任务切小步
→ 实现可验证
→ 问题能回流
→ 过程有留痕
```

---

## 3. 核心原则

### 原则一：不清不做

当需求无法回答关键问题时，不进入开发执行。此时应先做需求发散、边界澄清和验收标准收敛。

### 原则二：做必切小

进入开发前，必须把目标拆成能独立推进、独立验证、独立记录的小任务。

### 原则三：步步可验

每完成一个小任务，都应立即执行对应验证。验证失败时，不继续叠加新任务。

### 原则四：问题回流

发现偏差时，先判断它是 Bug、需求遗漏还是新需求，再决定修复、补文档或变更评估。

### 原则五：版本留痕

每次有意义的推进都应留下证据：文档、测试结果、变更记录或 Git 提交。

---

## 4. 适用边界

### 适用场景

* 新功能开发；
* 旧功能重构；
* Bug 修复；
* 多文件改动；
* 前后端联动；
* API、页面、模块、脚本、Skill 或 Agent 工作流开发；
* 需要测试、验收、提交记录和文档同步的任务。

### 不适合过度使用的场景

* 一行文案修改；
* 临时实验代码；
* 纯解释型问题；
* 不需要维护和验证的一次性片段；
* 用户明确要求快速草稿且不要求交付质量的场景。

---

## 5. 复杂度裁剪

流程强度必须匹配任务复杂度。

| 复杂度 | 适用任务 | 推荐流程 |
| --- | --- | --- |
| 轻量任务 | 单点 Bug、文案、小脚本、小样式调整 | 明确目标 → 快速执行 → 快速验证 → 简要留痕 |
| 中型任务 | 单个模块、一个 API 子系统、一个页面流程、一个 Skill | 需求校验 → 必要文档 → 任务拆分 → 小步实现 → 分层测试 → 修复与提交 |
| 复杂项目 | 多角色、多模块、长周期、涉及权限/数据/部署的系统 | 现状识别 → 发散澄清 → PRD/Spec → 架构设计 → 任务树 → 模块开发 → 阶段验收 → 系统测试 → Bug/变更回流 → 沉淀归档 |

判断标准：

* 是否影响多个文件或模块；
* 是否涉及数据结构、权限、安全、部署或迁移；
* 是否需要前后端联动；
* 是否存在多个用户角色或业务场景；
* 是否需要长期维护；
* 是否必须形成验收记录或版本记录。

## 6. 阶段入口

软件开发任务可以从任意阶段进入，但进入后必须检查前置条件。

| 入口 | 用户典型说法 | 第一动作 | 可能回流 |
| --- | --- | --- | --- |
| 需求澄清 | “我想做一个功能，但还没想清楚” | 发散分析并收敛需求 | 无需回流，当前就是起点 |
| 文档确认 | “PRD 已经有了，帮我看能不能开发” | 检查需求文档是否可开发 | 回到需求澄清 |
| 任务拆解 | “帮我把这个需求拆任务” | 拆成可验证小任务 | 回到文档补充 |
| 开发执行 | “按这个方案开始写代码” | 校验需求、任务和测试方式 | 回到文档或任务拆解 |
| Bug 修复 | “这个地方报错了” | 复现、归类、定位 | 回到需求或变更评估 |
| 测试验收 | “帮我检查是否完成” | 对照需求建立验收清单 | 回到 Bug 修复或任务补齐 |
| 留痕沉淀 | “帮我整理提交和记录” | 对齐代码、文档、测试与 Git | 回到测试或文档补齐 |

通用判断：

```text
识别入口阶段
→ 检查前置条件
→ 缺什么补什么
→ 执行当前阶段
→ 验证产出
→ 继续推进或回流
```

## 7. 三个核心门禁

### Gate 0：需求是否明确

开发前至少要能回答：

1. 用户是谁？
2. 要解决什么问题？
3. 输入是什么？
4. 输出是什么？
5. 成功标准是什么？
6. 不做什么？
7. 有哪些异常情况？
8. 涉及哪些页面、接口、数据、权限或文件？
9. 如何测试？
10. 如何判断完成？

需求状态分三类：

| 状态 | 含义 | 动作 |
| --- | --- | --- |
| CLEAR | 关键问题基本可回答 | 进入 Spec 或任务拆解 |
| PARTIAL | 有方向但缺少边界或验收 | 列假设和开放问题，补足关键项 |
| VAGUE | 目标、用户、输入输出或成功标准不清 | 不开发，先做需求发散与收敛 |

### Gate 1：需求文档是否可开发

中型及以上任务，需求文档至少应包含：

* 背景与目标；
* 用户角色；
* 核心场景；
* 功能范围；
* 非功能要求；
* 页面、API、模块或数据边界；
* 验收标准；
* 不做事项；
* 风险与开放问题。

### Gate 2：任务是否足够小

每个小任务应满足：

* 能在一次 focused session 内完成；
* 有明确输入和输出；
* 有验收标准；
* 有测试方式；
* 文件影响范围可控；
* 能独立提交或独立记录；
* 不混入无关变更。

---

## 8. 需求发散与收敛

当需求不明确时，不能只做简单追问，而应进行结构化发散。

如果必须先写文档，应显式标记不确定项，而不是替用户猜答案。推荐使用：

```text
[NEEDS CLARIFICATION: 这里写具体问题]
```

这类标记必须在进入实现前清理，或者转化为明确假设并写入风险。

### 发散维度

* 用户角色；
* 业务目标；
* 核心工作流；
* 输入与输出；
* 数据模型；
* 页面状态；
* API 边界；
* 权限与安全；
* 异常与失败场景；
* 部署、成本和维护；
* 未来扩展；
* 不做事项。

### 收敛产出

需求发散后必须收敛为：

* 需求本质；
* Must Have；
* Should Have；
* Later；
* Out of Scope；
* 验收标准；
* 风险；
* 开放问题。

推荐格式：

```markdown
## 需求收敛

### 需求本质
- ...

### Must Have
- ...

### Should Have
- ...

### Later
- ...

### Out of Scope
- ...

### 验收标准
- ...

### 开放问题
- ...
```

## 9. Spec 文档标准

中型及以上任务应创建或更新 Spec。根据项目结构，可使用：

```text
docs/requirements.md
docs/spec.md
docs/acceptance.md
docs/tasks.md
```

也可以在已有项目规范下使用等价文件。

### Spec 必备内容

```markdown
# 功能 / 项目名称

## 1. 背景

## 2. 目标

## 3. 用户角色

## 4. 核心场景

## 5. 功能范围

## 6. 非目标

## 7. 页面 / API / 模块边界

## 8. 数据结构

## 9. 异常与边界情况

## 10. 非功能要求

## 11. 验收标准

## 12. 测试策略

## 13. 风险与开放问题
```

### 用户故事标准

当功能涉及用户流程时，Spec 应按优先级组织用户故事：

```markdown
## User Story 1 - 标题 (Priority: P1)

### 用户价值
- ...

### 独立测试
- 只实现这个故事时，如何单独验证它已经可用？

### 验收场景
1. Given ...
   When ...
   Then ...
```

要求：

* P1 必须能构成最小可交付 MVP；
* 每个用户故事都应能独立开发、独立测试、独立演示；
* 验收场景优先使用 Given / When / Then；
* 不确定项必须标记为 `NEEDS CLARIFICATION`；
* Spec 聚焦“用户需要什么”和“为什么”，避免过早写实现细节。

Spec 不是写完就丢的说明书，而是后续任务拆解、验收测试和变更回流的共同依据。

## 10. 技术方案与实现计划

进入开发前，应形成足够轻量但明确的实现计划。

计划至少包含：

* 模块顺序；
* 依赖顺序；
* 涉及文件；
* 需要新增或修改的测试；
* 关键风险；
* 回滚点；
* 暂不处理的事项。

推荐格式：

```markdown
## 实现计划

### 模块顺序
1. ...

### 文件影响范围
- ...

### 测试计划
- ...

### 风险
- ...

### 回滚点
- ...
```

---

## 11. 任务拆解标准

任务拆解的目标不是把列表写长，而是让每个任务都能独立推进、验证和留痕。

### 合格任务卡

```markdown
## Task T001 [P] [US1]: ...

### 目标
- ...

### 范围
- 修改：
- 不修改：

### 验收标准
- ...

### 验证方式
- 命令：
- 手动检查：

### 影响文件
- ...

### 预期提交信息
- ...
```

字段说明：

* `T001`：稳定任务编号，便于追踪；
* `[P]`：可并行任务，仅在不同文件、无依赖时标记；
* `[US1]`：映射到具体用户故事；
* `验收标准`：说明完成后必须为真的条件；
* `验证方式`：写出具体命令或手动检查步骤；
* `影响文件`：尽量写真实路径，不写泛泛模块名。

### 拆分方式

可按以下维度拆分：

* 用户流程；
* 系统模块；
* 数据流；
* 前后端边界；
* 风险优先级；
* MVP 与增强项；
* 可独立验收的垂直切片。

### 任务阶段建议

复杂功能可以按以下阶段组织：

1. Setup：项目结构、依赖、配置；
2. Foundation：数据库、认证、路由、错误处理等阻塞性基础设施；
3. User Story P1：MVP 用户故事；
4. User Story P2 / P3：后续独立故事；
5. Polish：文档、性能、安全、无障碍、清理。

Foundation 阶段完成前，不应启动依赖它的用户故事开发。

## 12. 小步实现循环

每个小任务采用同一套执行循环：

```text
读取相关代码与文档
→ 明确当前小任务
→ 只改必要范围
→ 运行局部验证
→ 修复即时错误
→ 记录结果
→ 必要时提交
```

执行规则：

1. 一次只解决一个明确问题；
2. 不在一个小任务里混入无关重构；
3. 复杂问题先做验证性实现，再做完整实现；
4. 发现前提不成立时立即回流，不继续堆代码；
5. 当前任务验证失败时，不进入下一个任务；
6. 若必须临时扩大范围，应更新任务说明或 Spec。

## 13. 分层测试

测试应从小到大推进。

| 层级 | 目标 | 常见方式 |
| --- | --- | --- |
| 单点验证 | 验证函数、组件、接口、脚本 | 单元测试、局部命令、最小复现 |
| 小模块测试 | 验证独立功能片段 | 模块测试、组件测试、API 测试 |
| 大功能集成测试 | 验证完整业务链路 | 集成测试、端到端流程 |
| 前端整体测试 | 验证页面状态、交互、布局 | 浏览器检查、截图、Playwright |
| 后端整体测试 | 验证 API、数据、权限、错误处理 | API 测试、数据库检查、日志 |
| 全项目回归 | 验证项目未被破坏 | test / lint / build |
| 需求验收 | 对照 Spec 查漏补缺 | 验收清单、手动走查 |

测试记录应说明：

* 运行了什么；
* 结果是什么；
* 失败原因是什么；
* 是否已修复；
* 还有哪些风险。

---

## 14. 验收清单

项目或功能完成前，应对照需求文档建立验收清单。

```markdown
## 验收清单

### 已完成
- [x] ...

### 未完成
- [ ] ...

### Bug
- ...

### 需求遗漏
- ...

### 新需求
- ...

### 风险
- ...
```

验收不是只看代码是否运行，而是确认：

* 是否满足 Must Have；
* 是否没有误做 Out of Scope；
* 是否覆盖主要异常场景；
* 是否存在 UI、API、数据或权限偏差；
* 是否需要更新文档或任务清单。

### 验收三维度

对中型及以上功能，验收应至少覆盖三类问题：

| 维度 | 检查内容 |
| --- | --- |
| 完整性 | 任务是否完成，需求是否都有实现，验收场景是否覆盖 |
| 正确性 | 实现是否符合 Spec 意图，边界和错误路径是否正确 |
| 一致性 | 代码结构、命名、设计决策是否与计划和项目约定一致 |

---

## 15. Bug 与变更回流

发现问题时，先归类：

| 类型 | 含义 | 处理方式 |
| --- | --- | --- |
| Bug | 与既定需求不符 | 复现、定位、修复、测试、提交 |
| 需求遗漏 | 原需求应覆盖但未写清 | 回流补文档，再修正实现 |
| 新需求 | 新增目标或范围 | 做变更评估，决定是否进入当前版本 |

标准动作：

```text
描述问题
→ 复现问题
→ 判断归类
→ 修复或回流
→ 验证结果
→ 更新文档与记录
```

变更评估至少回答：

* 是否属于当前版本目标；
* 是否影响已有验收标准；
* 是否需要改数据、API、权限或 UI；
* 是否会扩大测试范围；
* 是否应进入 Later。

---

## 16. Git 证据链

Git 不只是保存最终代码，而是记录推进过程。

### 提交规则

* 一个小任务尽量对应一个提交；
* 不把无关文件混入提交；
* 提交信息应能看出对应需求、Bug 或任务；
* 文档变化与代码变化应保持一致；
* 新需求应先更新文档或任务清单，再实现；
* 不隐藏失败测试，也不把失败当成通过；
* 不提交密钥、账号、token、本地缓存和无关生成物。

### 推荐提交信息

```text
Add task breakdown for export workflow
Implement login validation error states
Fix article slug resolver edge case
Update acceptance checklist for billing flow
```

### 每轮结束应记录

* 完成了什么任务；
* 修改了哪些文件；
* 运行了哪些测试；
* 测试结果如何；
* 是否有提交；
* 剩余 Bug 或风险；
* 下一步是什么。

---

## 17. 标准完整流程

完整路径如下：

```text
输入需求
→ Gate 0：需求明确性判断
  ├─ CLEAR：进入 Spec / 任务拆解
  ├─ PARTIAL：列假设并补关键缺口
  └─ VAGUE：需求发散与收敛
→ Gate 1：需求文档是否可开发
→ 技术方案 / 实现计划
→ Gate 2：任务是否足够小
→ 小任务实现
→ 小任务验证
→ 小任务提交或记录
→ 大功能集成测试
→ 前端 / 后端整体测试
→ 项目级验收
→ Bug / 新需求 / 需求遗漏回流
→ 文档、提交与经验沉淀
```

该流程可裁剪，但不可跳过与任务风险相匹配的关键门禁。

## 18. 变更生命周期

对中型及以上需求，建议把“当前系统事实”和“待实施变更”分开管理：

```text
specs/      当前系统行为的事实来源
changes/    待实施或正在实施的变更
```

一个变更可以包含：

```text
changes/<change-name>/
├── proposal.md   # 为什么要改，改什么，不改什么
├── design.md     # 关键设计、取舍、风险
├── specs/        # 对现有能力的需求增量或修订
└── tasks.md      # 可执行任务清单
```

### 什么时候更新现有变更

适用于：

* 目标没有变，只是执行方式调整；
* Scope 收窄，例如先做 MVP；
* 实现中发现代码现状和预期不同；
* 验收标准需要补充，但仍服务同一目标。

### 什么时候新建变更

适用于：

* 意图已经改变；
* 范围扩大到另一个独立目标；
* 原需求可以单独完成和归档；
* 继续塞进原变更会让审查和回滚变困难。

### 完成与归档

变更完成前，应检查：

* `tasks.md` 是否完成或明确剩余项；
* Spec 中的需求是否有对应实现；
* 验收场景是否有测试或手动验证；
* `design.md` 中的关键决策是否反映在代码里；
* 若实现偏离设计，应更新设计文档或补充说明；
* 归档时将稳定需求合并回当前事实文档。

---

## 19. Agent 执行规范

当该模型用于 AI Agent 或 Codex 类工具时，执行者应遵守：

1. 先判断当前入口阶段，不默认从头开始；
2. 先检查前置条件，不在模糊需求下直接开发；
3. 明确本轮最小可验证动作；
4. 修改前阅读相关代码和文档；
5. 修改范围保持克制；
6. 每轮执行后做验证；
7. 发现需求问题时回流，不硬写代码；
8. 最终报告必须包含变更、验证、风险和下一步。

最终报告建议格式：

```markdown
## 完成内容
- ...

## 修改文件
- ...

## 验证
- 命令：
- 结果：

## 剩余风险
- ...

## 下一步
- ...
```

如果沉淀为 Skill，可命名为：

```yaml

tags:
  - software-development
  - spec-driven-development
  - incremental-implementation
  - layered-testing
  - git-workflow
```

Skill 的核心动作不应写成固定 Step 1 到 Step N，而应写成状态驱动流程：

```text
判断当前入口阶段
→ 检查前置条件
→ 按复杂度选择流程强度
→ 执行当前最小可验证动作
→ 验证结果
→ 判断继续、回流或结束
→ 留痕
```

## 22. 总结

“小步快跑 · 软件开发推进模型”的重点不是让开发变慢，而是避免在错误方向上投入太多。

它要求开发过程始终围绕五件事展开：

* 需求是否清楚；
* 任务是否足够小；
* 实现是否可验证；
* 问题是否能回流；
* 过程是否有证据链。

> **不清不做，做必切小；小步实现，步步可验；问题回流，版本留痕。**
