# 小步快跑 · 软件开发推进模型 v1.0

## 0. 文档定位

**中文名称：** 小步快跑 · 软件开发推进模型

**英文名称：** Software Delivery Model under Small Pace, Run Fast

**文档版本：** v1.0

**文档性质：** 软件开发流程文档 / AI 协作开发规范 / Skill 设计基础

---

## 1. 一句话定义

> **小步快跑 · 软件开发推进模型，是一套从任意阶段切入、以需求明确性为门禁、以小任务实现和分层验证为推进方式，并通过 Git 与文档形成证据链的 AI 协作开发方法。**

它不是固定的线性 SOP，而是“小步快跑”方法论在软件开发场景中的落地模型。

---

## 2. 核心目标

该模型要解决的是 AI 协作开发中常见的几类问题：

* 需求还不清楚就直接写代码；
* 一次性生成大块代码，难以验证和回滚；
* 小任务完成后没有测试，大功能集成时才集中暴露问题；
* Bug、需求遗漏和新需求混在一起处理；
* 代码、文档、测试结果和 Git 提交之间缺少对应关系；
* 最终交付无法追溯每一步为什么这样做。

因此，该模型强调：

```text
需求过门禁
→ 任务切小步
→ 实现可验证
→ 问题能回流
→ 过程有留痕
```

---

## 3. 适用场景

适用于：

* 新功能开发；
* 旧功能重构；
* Bug 修复；
* 多文件改动；
* 前后端联动；
* API、页面、模块、脚本、Skill 或 Agent 工作流开发；
* 需要测试、验收、提交记录和文档同步的任务。

不适合过度使用于：

* 一行文案修改；
* 临时实验代码；
* 纯解释型问题；
* 不需要维护和验证的一次性片段。

---

## 4. 阶段入口

软件开发任务可以从不同阶段进入：

| 入口 | 用户典型说法 | 应执行的第一动作 |
| --- | --- | --- |
| 需求澄清 | “我想做一个功能，但还没想清楚” | 发散分析并收敛需求 |
| 文档确认 | “PRD 已经有了，帮我看能不能开发” | 检查需求文档是否可开发 |
| 任务拆解 | “帮我把这个需求拆任务” | 拆成可验证的小任务 |
| 开发执行 | “按这个方案开始写代码” | 校验前置条件后小步实现 |
| Bug 修复 | “这个地方报错了” | 复现、归类、定位、修复 |
| 测试验收 | “帮我检查是否完成” | 对照需求建立验收清单 |
| 留痕沉淀 | “帮我整理提交和记录” | 对齐代码、文档、测试与 Git |

无论从哪里进入，都要先判断当前阶段的前置条件是否满足；不满足时应回流补齐。

---

## 5. 三个核心门禁

### Gate 0：需求是否明确

开发前至少要能回答：

1. 用户是谁？
2. 要解决什么问题？
3. 输入是什么？
4. 输出是什么？
5. 成功标准是什么？
6. 不做什么？
7. 有哪些异常情况？
8. 涉及哪些页面、接口、数据、权限或文件？
9. 如何测试？
10. 如何判断完成？

如果答不上来，应先进入需求发散与收敛，而不是直接开发。

### Gate 1：需求文档是否可开发

中型及以上任务，需求文档至少应包含：

* 背景与目标；
* 用户角色；
* 核心场景；
* 功能范围；
* 非功能要求；
* 页面、API、模块或数据边界；
* 验收标准；
* 不做事项；
* 风险与开放问题。

### Gate 2：任务是否足够小

每个小任务应满足：

* 能在一次 focused session 内完成；
* 有明确输入和输出；
* 有验收标准；
* 有测试方式；
* 文件影响范围可控；
* 能独立提交或独立记录；
* 不混入无关变更。

---

## 6. 标准推进流程

典型完整路径如下：

```text
需求明确性判断
→ 需求发散与收敛
→ 需求 / 方案文档
→ 技术方案
→ 任务拆解
→ 小任务实现
→ 小任务验证
→ 小任务提交
→ 大功能集成测试
→ 前端 / 后端整体测试
→ 项目级验收
→ Bug / 新需求 / 需求遗漏回流
→ 文档、提交与经验沉淀
```

这只是完整路径，不是强制顺序。简单任务可以裁剪，复杂任务应保留关键门禁。

---

## 7. 分层测试

测试应从小到大推进：

1. 单点验证：函数、组件、接口或脚本；
2. 小模块测试：独立功能片段；
3. 大功能集成测试：完整业务链路；
4. 前端整体测试：页面状态、交互、布局、浏览器行为；
5. 后端整体测试：API、数据、权限、错误处理；
6. 全项目构建或回归测试；
7. 对照需求文档做验收测试。

不应在小任务未验证的情况下连续堆叠多个改动。

---

## 8. Bug 与变更回流

发现问题时，先归类：

| 类型 | 含义 | 处理方式 |
| --- | --- | --- |
| Bug | 与既定需求不符 | 复现、定位、修复、测试、提交 |
| 需求遗漏 | 原需求应覆盖但未写清 | 回流补文档，再修正实现 |
| 新需求 | 新增目标或范围 | 做变更评估，决定是否进入当前版本 |

标准动作：

```text
描述问题
→ 复现问题
→ 判断归类
→ 修复或回流
→ 验证结果
→ 更新文档与记录
```

---

## 9. Git 证据链

Git 不只是保存最终代码，而是记录推进过程。

建议规则：

* 一个小任务尽量对应一个提交；
* 不把无关文件混入提交；
* 提交信息应能看出对应需求、Bug 或任务；
* 文档变化与代码变化应保持一致；
* 测试结果应能在开发记录或最终回复中说明；
* 新需求应先更新文档或任务清单，再实现；
* 不隐藏失败测试，也不把失败当成通过。

---

## 10. Skill 化方向

如果沉淀为 Skill，可命名为：

```yaml
name: small-pace-run-fast-development
display_name: 小步快跑 · 软件开发推进模型
description: >
  用于从模糊需求到明确需求文档，再到任务拆解、小步实现、
  分层测试、Bug 回流、新需求评估和 Git 证据链沉淀的 AI 协作开发流程。
```

Skill 的核心动作不应写成固定步骤，而应写成：

```text
判断当前入口阶段
→ 检查前置条件
→ 按复杂度选择流程强度
→ 执行当前最小可验证动作
→ 验证结果
→ 判断继续、回流或结束
→ 留痕
```

---

## 11. 总结

“小步快跑 · 软件开发推进模型”的重点不是让开发变慢，而是避免在错误方向上投入太多。

它要求开发过程始终围绕五件事展开：

* 需求是否清楚；
* 任务是否足够小；
* 实现是否可验证；
* 问题是否能回流；
* 过程是否有证据链。

> **不清不做，做必切小；小步实现，步步可验；问题回流，版本留痕。**
