
可以进入，但建议**不要一上来就 fork 或重写 Pi / oh-my-pi**。更稳的路径是：先把 SmartDev Agent 做成一个**可插拔的开发型 Agent 外壳**，用 Pi / oh-my-pi 的设计作为参考，先实现你自己的核心流程、项目适配器和 Skill 调用机制。

# 1. 先看这两个仓库给我们的启发

## Pi 的价值

Pi 更像一个“Agent Harness 基座”。它的 monorepo 里明确拆出了：

* `pi-coding-agent`：交互式编码 Agent CLI
* `pi-agent-core`：带工具调用和状态管理的 Agent runtime
* `pi-ai`：统一多模型 Provider API
* `pi-tui`：终端 UI 库 

这对 SmartDev Agent 很关键。说明一个成熟编码 Agent 不应该只是一个 Prompt，而应该至少拆成：

```txt
Agent Core
Tool / Skill Runtime
Model Provider
CLI / UI Surface
Project Rules
```

Pi 还很强调项目级规则，比如 `AGENTS.md` 里要求宽泛修改前必须读完整文件、改代码后运行 `npm run check`、不要擅自 commit、Git 操作不能破坏其他 session 的工作等。
这和你 SmartDev Agent 的“先诊断、后修改、小步快跑、每步可验证、不扩大范围”高度一致。

## oh-my-pi 的价值

oh-my-pi 更像“加强版产品化编码 Agent”。它的 README 明确强调 **IDE wired in**，并列出 LSP、DAP、subagents、memory、skills、internal schemes、browser 等能力；它还提到工具表面统一成类似 filesystem 的读写方式，以及 `skill://`、`agent://`、`pr://` 等内部 scheme。

它的 `package.json` 也显示这是一个 Bun monorepo，并包含 `pi-agent-core`、`pi-ai`、`pi-coding-agent`、`pi-tui`、`pi-natives`、`pi-utils` 等包。

对 SmartDev 的启发是：

```txt
Skill 不应该只是“提示词文件”
Skill 应该像一种可发现、可调用、可验证的工具资源
```

# 2. 你的 SmartDev Agent 应该怎么定位

结合 Pi / oh-my-pi 后，我建议 SmartDev 不要定位为“另一个通用编码 Agent”，而是定位为：

> **面向真实项目推进的项目开发 Agent：比普通 coding agent 更重视项目诊断、任务拆解、设计规范、风险控制、文档沉淀和验收闭环。**

也就是说：

| 类型               | 重点                                      |
| ---------------- | --------------------------------------- |
| Pi / oh-my-pi    | Agent harness、工具调用、编码能力、TUI、LSP/DAP、运行时 |
| SmartDev Agent   | 项目治理、任务拆解、适配器、文档边界、开发流程、验收标准            |
| SmartFav Adapter | SmartDev 的第一个真实项目适配器                    |

这能避免和 Pi / oh-my-pi 正面重复造轮子。

# 3. 三种进入 Agent 开发的路线

## 路线 A：基于 Pi / oh-my-pi 直接改

优点：

* 起步有现成 Agent runtime。
* 工具调用、TUI、Provider、会话、规则体系都比较成熟。
* oh-my-pi 还包含 LSP、DAP、subagents、memory、skills 等较完整能力。

缺点：

* 项目复杂度高。
* 技术栈偏 TypeScript / Bun / Rust。
* 一上来 fork，容易被原项目架构牵着走。
* 你的 SmartDev 目标会被淹没在大型编码 Agent 的细节里。

**不建议第一步就走这条。**

---

## 路线 B：参考 Pi / oh-my-pi，自己做 SmartDev MVP

这是我最推荐的。

先做一个轻量版：

```txt
SmartDev CLI
├── Agent Core
├── Skill Registry
├── Project Adapter Loader
├── Risk Controller
├── Context Manager
└── Reporter
```

先不做 TUI、不做 LSP、不做 DAP、不做浏览器、不做 subagents。

第一阶段只实现：

```txt
repo.scan
repo.diagnose
architecture.map
token.audit
task.plan
risk.check
qa.checklist
change.summary
```

这些 Skill 大多是只读分析，不容易误伤项目。

优点：

* 和 SmartDev 的真实目标一致。
* 不会陷入大型 Agent 工程复杂度。
* 可以先服务 SmartFav。
* 之后再决定是否接入 Pi / oh-my-pi 的能力。

---

## 路线 C：先作为 Pi / oh-my-pi 的 Skill Pack

这个也可以，但建议放在第二阶段。

比如 SmartDev 不先做完整 Agent，而是做成一组技能：

```txt
skill://smartdev/repo-diagnose
skill://smartdev/token-audit
skill://smartdev/task-plan
skill://smartdev/project-adapter
skill://smartdev/qa-checklist
```

oh-my-pi 已经有“walk skills”和 `skill://` 这类内部 scheme 的思路。

但问题是：你需要先摸清它的 skill 实现方式和包结构。否则容易又变成“研究别人的大型项目”，而不是推进自己的 Agent。

# 6. Skill 调用应该怎么设计

Skill 必须是结构化的，不要只是 Prompt。

每个 Skill 至少包含：

```yaml
id: token.audit
name: Design Token Audit
capability: token_governance
risk: R0
mode: read_only
inputs:
  - project_path
  - adapter
outputs:
  - token_sources
  - duplicated_tokens
  - hardcoded_colors
  - coverage_rate
  - recommendations
```

## Skill 分三类

| 类型       | 示例                                           | 是否可自动执行 |
| -------- | -------------------------------------------- | ------- |
| 只读 Skill | `repo.scan`、`token.audit`、`architecture.map` | 可以      |
| 生成 Skill | `doc.generate`、`task.plan`、`qa.checklist`    | 可以生成草案  |
| 修改 Skill | `code.patch`、`file.write`、`refactor.apply`   | 必须走风险控制 |

第一阶段只做前两类。

# 7. 借鉴 Pi / oh-my-pi 的几个关键设计

## 7.1 Agent Core 与 Coding Agent 分离

Pi 将 `pi-agent-core` 和 `pi-coding-agent` 分开，说明“Agent runtime”和“具体编码应用”不应该混在一起。

SmartDev 也应该这样：

```txt
smartdev-core       # 状态机、Skill 调用、风险控制
smartdev-cli        # 命令行入口
smartdev-skills     # 技能
smartdev-adapters   # 项目适配器
```

---

## 7.2 项目规则必须显式化

Pi / oh-my-pi 都大量依赖 `AGENTS.md` 这类项目规则。Pi 的规则明确要求改动前读完整文件、不要猜外部 API 类型、不要擅自删除功能、不要随便 commit。

SmartDev 也应支持：

```txt
AGENTS.md
SMARTDEV.md
.smartdev/project.yaml
```

其中 `SMARTDEV.md` 可以写：

```txt
项目目标
技术栈
禁止修改区域
验证命令
文档更新规则
当前优先任务
```

---

## 7.3 Tool / Skill 必须有 preflight

Pi agent-core 里有 `beforeToolCall`、`afterToolCall`、tool execution mode 等机制。

SmartDev 应该内置：

```txt
beforeSkillCall
afterSkillCall
riskCheck
adapterCheck
scopeCheck
```

例如：

```txt
code.patch 执行前：
- 是否属于当前任务范围？
- 是否改了禁止路径？
- 是否 R2/R3？
- 是否需要用户确认？
- 是否有回滚方案？
```

---

## 7.4 不要跳过验证

Pi 的开发规则要求代码修改后运行检查命令，并且明确不要随便运行完整测试或 build，除非用户要求。

SmartDev 也要这样：

```txt
不同项目 Adapter 定义不同验证命令
Agent 不自己乱跑重型命令
先推荐最小验证
必要时请求确认
```

例如 SmartFav：

```yaml
verify:
  extension:
    - "Chrome 加载扩展"
    - "Service Worker 无报错"
    - "Side Panel 可打开"
  server:
    - "uvicorn apps.server.main:app --reload"
    - "GET /api/items"
  ui:
    - "320px / 400px / 480px 检查"
```

# 8. SmartDev MVP 的具体开发入口

不要先做大而全。第一版 CLI 就 4 个命令够了：

```bash
smartdev diagnose /path/to/project
smartdev audit-token /path/to/project
smartdev plan /path/to/project --task "统一 tokens.css"
smartdev verify-plan /path/to/project --scope sidepanel
```

## 命令 1：diagnose

调用：

```txt
repo.scan
repo.diagnose
architecture.map
```

输出：

```txt

技术栈
目录结构
文档状态
风险点
下一步建议
```

## 命令 2：audit-token

调用：

```txt
token.audit
```

输出：

```txt
tokens.css 来源
重复 token
硬编码颜色
覆盖率
替换建议
```

## 命令 3：plan

调用：

```txt
task.plan
risk.check
qa.checklist
```

输出：

```txt
任务拆解
影响文件
风险等级
执行顺序
验收标准
```

## 命令 4：verify-plan

调用：

```txt
adapter.validation
qa.checklist
```

输出：

```txt
针对项目类型的验收清单
```

# 9. 技术栈怎么选

这里有两个选择。

## 方案 A：Python 实现 SmartDev MVP

适合你当前路线。

优点：

* 快速实现文件扫描、正则检查、文档生成。
* 和你现有 Document Factory / Image Factory / FastAPI 经验一致。
* 后续能和本地工具、文档转换、数据处理结合。

缺点：

* 如果要做 TUI、LSP、DAP，后续会吃力。

适合第一阶段。

---

## 方案 B：TypeScript / Bun 实现

更接近 Pi / oh-my-pi。

优点：

* 更容易参考 Pi / oh-my-pi 的设计。
* 对前端/插件/Node 项目分析更自然。
* Bun 的工具链适合命令行 Agent。

缺点：

* 初期复杂度高。
* 会被 oh-my-pi 的工程模式影响较多。

适合第二阶段。

---

## 我的建议

第一版用 Python。

原因：

```txt
SmartDev 第一阶段重点是“项目诊断 + 文档治理 + 任务拆解”
不是 LSP / DAP / TUI / subagent
```

等 SmartDev 的核心流程跑通后，再考虑：

```txt
TypeScript 版 Skill Runtime
或
接入 oh-my-pi 作为底层 coding harness
```

暂时不建议。

更好的策略是：

```txt
先 clone 阅读
再抽象设计
不 fork 开发
不直接改它们
```

可以把它们当作参考对象：

| 参考对象     | 学什么                                                          |
| -------- | ------------------------------------------------------------ |
| Pi       | Agent core、tool calling、state management、package 分层          |
| oh-my-pi | 工具表面统一、LSP/DAP、subagents、memory、skill scheme、IDE integration |
| SmartDev | 项目治理、适配器、任务拆解、风险控制、文档沉淀                                      |

真正需要 fork 的时机是：

```txt
你已经证明 SmartDev 的流程有效
并且明确需要 Pi / oh-my-pi 的 runtime 能力
```

否则 fork 只会带来维护负担。

# 1. 这些系统分别值得参考什么

## Pi / oh-my-pi：参考 Agent Harness 与工具系统

Pi 的仓库已经把能力拆成几个包：`pi-coding-agent` 是交互式编码 Agent CLI，`pi-agent-core` 是带工具调用和状态管理的 Agent runtime，`pi-ai` 负责多模型 Provider，`pi-tui` 负责终端 UI。

oh-my-pi 在此基础上更产品化，强调 IDE wired in、LSP、DAP、subagents、memory、skills、内部 scheme、浏览器等能力。它的 README 提到 `skill://`、`agent://`、`pr://` 等内部资源 scheme，并把工具统一成类似文件系统的读写接口。

SmartDev 应该从这里学：

```txt
Agent Core 和 Coding Agent 分离
Tool / Skill 统一注册
文件、PR、Issue、Skill 都可以抽象成资源
工具调用要有事件流和状态管理
子 Agent 可以用于并行分析，但不能第一阶段就做
```

---

## Claude Code：参考权限、上下文、Skills、Hooks、Subagents

有研究把 Claude Code 的设计概括为：核心是“模型循环调用工具”，但真正复杂的是周边系统，包括权限系统、上下文压缩、MCP / plugins / skills / hooks、subagent delegation、append-only session storage 等。([arXiv][1])

对 SmartDev 有价值的是：

```txt
权限系统
上下文管理
技能机制
Hook 机制
子 Agent 分派
会话记录
```

但要注意：Claude Code 是通用编码 Agent，SmartDev 的目标不是复制它，而是把这些机制转化成**项目推进型 Agent**的能力。

SmartDev 可吸收为：

```txt
beforeSkillCall
afterSkillCall
permission gate
context compaction
project memory
execution transcript
```

---

## Codex：参考任务沙箱、变更可审查、命令日志

Codex 作为软件工程 Agent，关键特点是：每个任务在独立环境中执行，预加载用户仓库，可以读写文件、运行测试和调用代码检查工具；完成后返回结果供用户审查，并提供命令日志和测试结果。([维基百科][2])

这对 SmartDev 非常关键。

SmartDev 应该学：

```txt
任务隔离
每次任务有独立上下文
所有改动必须可审查
返回命令日志和测试结果
默认不直接污染主仓库
```

但第一阶段不必做真正沙箱，可以先做：

```txt
默认只读
默认生成 patch
用户确认后应用
每次应用前后输出 changed files / validation / rollback
```

---

## OpenClaw：参考长任务、自动化、消息通道，但重点学安全边界

OpenClaw 的价值在于它偏“个人自动化 Agent”：通过消息平台、任务调度、文件、外部服务连接来执行长期任务。公开资料将其描述为可运行本地、连接外部模型、通过消息平台作为主要交互界面的自治助手。([维基百科][3])

但 OpenClaw 也特别适合作为安全反例。近期开源研究分析 OpenClaw 的攻击面，指出 agent 框架连接 shell、文件系统、容器、消息通道和插件时，会产生跨层信任问题；研究还提到恶意 skill、exec allowlist 假设失效、gateway 与 host 层组合攻击等风险。([arXiv][4])

SmartDev 应该学：

```txt
长任务调度
消息入口
文件工作区
外部服务连接
插件 / Skill 分发
```

更重要的是吸取教训：

```txt
不能只靠每层各自校验
需要统一 Policy Engine
Skill 安装和执行必须分开授权
Shell / File / Network / Git 必须分权限
所有可执行 Skill 必须有审计日志
```

---

## Hermes：参考记忆、流程沉淀、任务状态

你之前提到的 Hermes，更接近你个人使用经验里的“记忆 / 流程沉淀 / Skills / Workflows / Dialogues / Archive”这类系统。公开搜索结果里也有“Portable Agent Memory”这类相关方向，强调跨 Agent 的结构化记忆、能力级访问控制和注入防护。([arXiv][5])

SmartDev 应该从 Hermes 方向学：

```txt
项目记忆
工作流沉淀
Skill 运行记录
任务状态
历史决策
复盘归档
```

但不要一开始做复杂 memory。SmartDev 的第一版记忆可以很简单：

```txt
.smartdev/memory/project.md
.smartdev/memory/decisions.md
.smartdev/memory/known-issues.md
.smartdev/runs/YYYYMMDD-HHMM-task.json
```

# 2. 对 SmartDev 的参考价值排序

如果按“现在进入开发”的优先级，我建议这样排：

| 优先级 | 参考对象          | 学什么                           | 现在是否实现        |
| --: | ------------- | ----------------------------- | ------------- |
|   1 | Codex         | 任务隔离、审查式交付、命令日志、测试结果          | 现在就吸收         |
|   2 | Claude Code   | 权限、上下文、skills、hooks、subagents | 先吸收权限和 skills |
|   3 | Pi / oh-my-pi | Agent core、工具调用、资源 scheme、TUI | 先参考，不重写       |
|   4 | OpenClaw      | 长任务、自动化、消息入口、安全边界             | 只吸收安全边界       |
|   5 | Hermes        | 项目记忆、流程沉淀、归档                  | 第二阶段再做        |

# 3. SmartDev 不应该照搬什么

## 不要一开始照搬 oh-my-pi 的全能力

oh-my-pi 很强，但能力面太大：LSP、DAP、subagents、browser、memory、skills、internal scheme、IDE integration。
SmartDev 如果第一阶段就追这些，会偏离核心目标。

第一阶段只要：

```txt
读项目
识别项目
加载适配器
调用 Skill
输出计划
输出验收
```

---

## 不要一开始照搬 OpenClaw 的自动化形态

OpenClaw 的多通道个人助手模式很有吸引力，但也放大了安全问题。尤其是连接 shell、插件、文件系统、消息平台后，攻击面会急剧扩大。([arXiv][4])

SmartDev 第一阶段不应该做：

```txt
自动定时任务
自动接消息平台
自动执行 shell
自动修改仓库
自动安装 Skill
```

---

## 不要把 Claude / Codex 的产品形态当成 SmartDev 的终点

Claude / Codex 是编码工具；SmartDev 的差异是：

```txt
项目治理
文档边界
任务拆解
设计令牌治理
执行协议
项目适配器
```

这个差异要保住。

# 5. Skill 调用机制要加“权限门”

这是从 OpenClaw、Claude、Codex 都能看到的共同点：**Agent 的核心不是能不能调用工具，而是能不能安全调用工具。**

SmartDev 的 Skill 可以分级：

| 权限         | 类型   | 示例                        | 默认   |
| ---------- | ---- | ------------------------- | ---- |
| READ       | 只读   | 扫描目录、读取文件、分析 token        | 自动允许 |
| PLAN       | 规划   | 生成任务、验收清单、patch 草案        | 自动允许 |
| WRITE_DOC  | 写文档  | 生成 README、更新 progress     | 需要确认 |
| WRITE_CODE | 改代码  | 应用 patch、修改 CSS/JS/Python | 需要确认 |
| EXEC       | 执行命令 | 运行 check、test、shell       | 需要确认 |
| NETWORK    | 网络访问 | 拉网页、访问 GitHub API         | 需要确认 |
| SECRET     | 密钥相关 | 读取 env、token、配置           | 默认禁止 |

这个应该直接放进 `smartdev-agent-core-spec.md` 的 Skill 调用机制里。

# 6. SmartDev 的第一阶段不要叫 Coding Agent

建议第一阶段叫：

> **SmartDev Diagnose Agent**

因为第一阶段真正要做的是：

```txt
项目诊断
架构分析
Token 检查
任务规划
风险判断
验收清单
```

而不是自动写代码。

等进入第二阶段，再升级为：

> **SmartDev Execution Agent**

支持：

```txt
patch 生成
patch 应用
局部验证
文档同步
变更总结
```

第三阶段才考虑：

> **SmartDev Multi-Agent Workflow**

支持：

```txt
子 Agent
并行分析
多项目适配器
长期记忆
IDE / TUI
```

# 7. 后续参考这些系统时，应该回答 5 个问题

不要泛泛“学习 Claude / Codex”。每次参考一个系统，只问：

```txt
1. 它如何表示项目上下文？
2. 它如何发现和调用工具 / Skill？
3. 它如何控制权限和风险？
4. 它如何记录执行过程？
5. 它如何把结果交给用户审查？
```

这 5 个问题就是 SmartDev Agent 的参考框架。

# 10. 最终判断

需要参考这些系统，但参考方式应该是：

```txt
Pi / oh-my-pi：学 Agent runtime 和工具表面
Claude Code：学权限、skills、hooks、subagents、上下文
Codex：学任务隔离、审查式交付、命令日志
OpenClaw：学自动化能力，也重点学安全边界
Hermes：学项目记忆、流程沉淀、任务归档
```

但 SmartDev 的第一性原则不能变：

> **它不是为了做一个万能 Agent，而是为了把真实项目推进得更清晰、更稳定、更可验证。**
